Toolkit versioning ensures your tools behave consistently across deployments. You can pin specific versions in production, test new releases in development, and roll back easily when needed.

## Quick start

### Method 2: Environment variables

Set toolkit versions through environment variables. Useful for deployment configurations.

```bash
# Set versions for specific toolkits
export COMPOSIO_TOOLKIT_VERSION_GITHUB="20250116_00"
export COMPOSIO_TOOLKIT_VERSION_SLACK="20250115_01"

# Or set one version for all toolkits
export COMPOSIO_TOOLKIT_VERSION="20250116_00"
```

## Version formats

### Version identifiers

Toolkit versions follow the format `YYYYMMDD_NN`:
- `YYYYMMDD`: Release date
- `NN`: Sequential release number for that day

```python
# Production: Use specific version
toolkit_versions = {
    "github": "20250116_00"  # Released January 16, 2025, first version
}

# Development: Use latest
toolkit_versions = {
    "github": "latest"  # Always use newest version
}
```

### Version resolution

Versions are resolved in this order:
1. **Per-execution version** (if specified)
2. **SDK initialization version** (if configured)
3. **Environment variable** for specific toolkit
4. **Global environment variable** (applies to all)
5. **Default**: `latest`

<Warning>
Never use `latest` in production. It can introduce breaking changes without warning.
</Warning>

</Warning>

### Migration checklist

1. **Audit current usage**: Identify all `tools.execute()` calls
2. **Choose strategy**:
   - Set toolkit versions in SDK initialization (recommended)
   - Add version parameter to each execute call
   - Use environment variables for deployment config
3. **Test in staging**: Verify tools work with pinned versions
4. **Deploy gradually**: Roll out to production incrementally
5. **Monitor errors**: Watch for version-related failures

## Best practices

### Version pinning strategy

```python
# Development: Latest for rapid iteration
dev_versions = {"github": "latest", "slack": "latest"}

# Staging: Test specific new versions
staging_versions = {"github": "20250116_00", "slack": "20250115_00"}

# Production: Pin stable versions
prod_versions = {"github": "20250110_00", "slack": "20250110_00"}
```

### Update workflow

1. **Development**: Test with `latest`
2. **Staging**: Pin specific version for testing
3. **Validation**: Run automated tests
4. **Production**: Deploy tested version
5. **Monitor**: Track errors and performance


## Reference

### SDK parameters

<CodeGroup>
```python Python
Composio(
    api_key="YOUR_API_KEY",
    toolkit_versions={
        "toolkit_name": "version"  # Dict[str, str]
    }
)
```

```typescript TypeScript
new Composio({
    apiKey: "YOUR_API_KEY",
    toolkitVersions: {
        toolkitName: "version"  // Record<string, string>
    }
})
```
</CodeGroup>

### Environment variables

```bash
# Specific toolkit version
COMPOSIO_TOOLKIT_VERSION_[TOOLKIT_NAME]="version"

# Global version (all toolkits)
COMPOSIO_TOOLKIT_VERSION="version"

# Examples
COMPOSIO_TOOLKIT_VERSION_GITHUB="20250116_00"
COMPOSIO_TOOLKIT_VERSION_SLACK="20250115_01"
COMPOSIO_TOOLKIT_VERSION="20250110_00"
```

### Error codes

| Error | Description | Solution |
|-------|-------------|----------|
| `TOOLKIT_VERSION_REQUIRED` | Manual execution without version | Specify toolkit_version parameter |
| `INVALID_TOOLKIT_VERSION` | Version format incorrect | Use YYYYMMDD_NN format |
| `VERSION_NOT_FOUND` | Specified version doesn't exist | Check available versions |
| `TOOLKIT_NOT_VERSIONED` | Toolkit doesn't support versioning | Contact support |
